Hey Folks 👋 , In this article we will explore **OSGi (Open Service Gateway Initiative)** — the foundational technology behind AEM's modular architecture. Understanding OSGi is essential before diving into services, components, or any serious AEM backend development.

OSGi enables component modularity while maintaining service-oriented communication patterns — allowing AEM to be built from dozens of independent, dynamically manageable modules.

## Why OSGi in AEM?

### Modularity
Applications can be decomposed into reusable, independent modules called **bundles** that support dynamic installation, updates, and removal without system restarts. AEM core components (like Title, Image, Button) are themselves implemented as OSGi bundles.

### Dynamic Management
OSGi automatically resolves dependencies and allows service updates without full application restarts. Services can be modified independently without impacting the broader system.

### Configuration Flexibility
Centralized configuration management through `/system/console/configMgr` enables administrators to adjust settings dynamically via `@Designate` annotations — without any code modifications.

### Improved Maintainability & Scalability
System decoupling enhances debugging and maintenance while supporting independent development, testing, and deployment of services across large projects.

### Service-Oriented Development
Encapsulating functionality into OSGi Services with dependency injection via `@Reference` and `@OSGiService` promotes clean, testable architectural patterns.

## OSGi Bundle Lifecycle Stages

Understanding the bundle lifecycle is key to diagnosing issues in the Felix console.

| Stage | Description |
|---|---|
| **INSTALLED** | Bundle loaded but dependencies unresolved |
| **RESOLVED** | Dependencies resolved; bundle inactive |
| **STARTING** | Dependencies ready; executing `@Activate` method |
| **ACTIVE** | Services available; bundle fully operational |
| **STOPPING** | Bundle halted; executing `@Deactivate` method |
| **UNINSTALLED** | Removed due to unresolved dependencies or failures |

You can monitor bundle states at:
```
http://localhost:4502/system/console/bundles
```

## Key OSGi R6 Annotations

| Annotation | Purpose |
|---|---|
| `@Component` | Designates a class as an OSGi component |
| `@Reference` | Injects a dependency (another OSGi service) |
| `@Activate` | Executes when the component loads and resolves |
| `@Deactivate` | Executes during component shutdown |
| `@Modified` | Triggers when the component's configuration is updated |
| `@Designate` | Links the class to a configuration interface |
| `@ObjectClassDefinition` | Defines configuration metadata for a component |
| `@AttributeDefinition` | Specifies individual configuration properties |

## A Simple OSGi Component Example

```java
@Component(service = MyService.class, immediate = true)
@Designate(ocd = MyService.Config.class)
public class MyServiceImpl implements MyService {

    @ObjectClassDefinition(name = "My Service Configuration")
    public @interface Config {
        @AttributeDefinition(name = "API URL", description = "External API endpoint")
        String apiUrl() default "https://api.example.com";
    }

    private String apiUrl;

    @Activate
    @Modified
    protected void activate(Config config) {
        this.apiUrl = config.apiUrl();
    }

    @Deactivate
    protected void deactivate() {
        // cleanup
    }

    @Override
    public String getApiUrl() {
        return apiUrl;
    }
}
```

> [!Note]
> Check the Felix console at `http://localhost:4502/system/console/configMgr` to see and modify your component's configuration at runtime without redeployment.
