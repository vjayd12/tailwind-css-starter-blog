Hey Folks 👋 , In this article we will explore **BYOM (Bring Your Own Markup)** — a powerful pattern gaining traction in Edge Delivery Services (EDS) projects. BYOM allows you to integrate externally generated HTML content into AEM Edge Delivery Services rather than relying exclusively on traditional AEM repositories or Google Drive/SharePoint.

This is the architecture you need when your content comes from third-party systems, external APIs, or services built outside of AEM.

## What is BYOM?

BYOM allows developers to call external endpoints that return HTML markup. The edge CDN fetches this HTML and serves it as part of your EDS page — transparently from the user's perspective. It enables flexible content sourcing beyond what the standard AEM content bus supports.

## Problems BYOM Solves

- **No Content Migration** — Existing content stays in external systems; no import needed
- **Dynamic Page Generation** — Pages driven by live API data, not static authored content
- **Integration Flexibility** — Connect to any backend that returns HTML
- **API-Driven Websites** — Build pages entirely from external data sources
- **Faster Development** — Frontend teams work independently of AEM content workflows

## Prerequisites

Before implementing BYOM, you need:

1. An EDS-based website up and running
2. Adobe CDN (Fastly) enabled for your domain
3. **Edge worker support** — requires an Adobe support ticket to enable
4. An edge worker project set up (using Fastly Compute or equivalent)

> [!Note]
> Edge worker support is not enabled by default. Submit an Adobe support ticket early in your project timeline as it may take time to provision.

## Architecture Overview

The request flow works like this:

```
User Request
    → CDN (Fastly/Edge)
    → Edge Delivery Services
    → Configured BYOM Endpoint
    → External Service (Node.js, Adobe App Builder, APIs)
    → Data Collection from Multiple Systems
    → HTML Generation
    → CDN Delivery to User
```

The key insight: the HTML generation happens at the edge, close to the user, not on a central origin server.

## Site Configuration

In your EDS project's `config.json`, add an `overlay` object alongside the standard content bus mapping:

```json
{
  "version": 1,
  "contentBus": {
    "content": {
      "source": "https://main--<repo>--<org>.hlx.page"
    }
  },
  "overlay": {
    "cards": {
      "source": "https://<your-edge-worker>.edgecompute.app"
    }
  }
}
```

The `overlay` object tells EDS to route requests matching the `cards` path pattern to your external service instead of the content bus.

## Implementation: Step-by-Step

This example builds a card list page and detail page using the JSONPlaceholder API.

**Step 1 :-** Create routes in your edge compute project for third-party content:

```javascript
// routes/cards.js
export async function handleCards(request) {
  const url = new URL(request.url);
  const id = url.searchParams.get('id');

  if (id) {
    return handleCardDetail(id);
  }
  return handleCardList();
}
```

**Step 2 :-** Build JavaScript functions that parse external data and generate HTML markup:

```javascript
async function handleCardList() {
  const response = await fetch('https://jsonplaceholder.typicode.com/posts?_limit=10');
  const posts = await response.json();

  const html = `
    <html>
    <body>
      <div class="cards-list">
        ${posts.map(post => `
          <div class="card">
            <h2>${post.title}</h2>
            <p>${post.body.substring(0, 100)}...</p>
            <a href="/cards?id=${post.id}">Read More</a>
          </div>
        `).join('')}
      </div>
    </body>
    </html>
  `;

  return new Response(html, {
    headers: { 'Content-Type': 'text/html' }
  });
}

async function handleCardDetail(id) {
  const response = await fetch(`https://jsonplaceholder.typicode.com/posts/${id}`);
  const post = await response.json();

  const html = `
    <html>
    <body>
      <article>
        <h1>${post.title}</h1>
        <p>${post.body}</p>
      </article>
    </body>
    </html>
  `;

  return new Response(html, {
    headers: { 'Content-Type': 'text/html' }
  });
}
```

**Step 3 :-** Configure service names in `compute.yaml`:

```yaml
name: byom-cards-service
language: javascript
main: src/index.js
```

**Step 4 :-** Write matching rules in `cdn.yaml` to route requests to your edge worker:

```yaml
rules:
  - name: byom-cards
    match:
      url: /cards*
    action:
      type: compute
      service: byom-cards-service
```

**Step 5 :-** Deploy the CDN rules and compute services via the Adobe CDN management tooling.

## When to Use BYOM

| Scenario | Recommendation |
|---|---|
| Content already in an external CMS | Use BYOM — avoid double-authoring |
| Product catalog from a PIM system | Use BYOM — keep PIM as source of truth |
| Marketing pages authored by editors | Use standard EDS content bus |
| Real-time data (prices, inventory) | Use BYOM — static content can't serve this |
| Blog or editorial content | Use standard EDS content bus |

## Conclusion

BYOM represents a significant architectural shift toward modular, externally-sourced content delivery within Adobe's edge infrastructure. It's the pattern that truly unlocks EDS for organizations with existing content systems they cannot or should not migrate.
